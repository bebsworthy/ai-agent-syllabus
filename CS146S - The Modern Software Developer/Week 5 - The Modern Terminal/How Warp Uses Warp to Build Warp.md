# How Warp Uses Warp to Build Warp

**Source:** https://notion.warp.dev/How-Warp-uses-Warp-to-build-Warp-21643263616d81a6b9e3e63fd8a7380c

## Warp's Coding Mandate

Every coding task should start with a prompt in Warp.

### The Flow

1. **If the coding task succeeds:** Share the success in #warped-it
2. **If after ten minutes you feel like you are wasting time by prompting:**
   - Share feedback with the relevant #feedback- channel on what failed
   - Share your prompt, conversation id, and other context so the team can learn from the issue and build evals around it
   - This is a chance for folks to give each other tips on how best to use Warp to code
3. **Try another AI coding tool** (e.g., Cursor, Claude) if you have more luck – report what went well and what didn't
4. **Code by hand** if needed – they need to actually ship software and if that's still fastest, you should do it

## Why This Mandate?

### Performance Benefits
- Coding in Warp by prompt is actually faster for many tasks and should increase overall engineering output
- Saves time from having to handwrite code
- Allows for multi-threaded development
- Especially useful when working on areas of code you are less familiar with – Warp can help you learn these new parts of the codebase quickly

### Dogfooding
- Dogfooding the product is the best way to improve it
- Since the team is asking users to work this way and betting their business on it, they need to believe enough in the workflow to use it themselves
- Using competitor products builds intuition on where Warp might improve and what is possible

## Guidelines for Coding with Warp

### Responsibility and Quality Standards
- You are responsible for the code you submit for review
- You must understand all code that you submit for review at the same level as if you had written it by yourself
- Any code generated by agent mode must be at the same quality expected for hand-written code:
  - Must be well-factored
  - Must be well-tested
  - Must follow coding conventions (comments, APIs, imports, test file location, etc.)
- It is never an excuse for a bug or poor code quality to say "AI wrote this" – AI is a tool, not a responsible party on the software team
- All regular coding guidelines still apply when coding by prompt (small PRs, feature flags, and so on)

### How to Prompt the Agent

**Don't just tell the agent what outcome you want – tell the agent how to engineer the change.**

If you just tell the agent "I want a feature that works like X," the agent is likely to produce code that isn't up to standards. Instead, tell the agent:
- What should the data model look like?
- What is the API you want?
- Where should the code live?
- What tests should it write?

Just letting the agent goal-seek based on the outcome will end up wasting a lot of time by producing code that may "work" but that isn't maintainable, testable, or composable.

### Breaking Down Tasks

**Break down coding tasks into smaller components – do not one-shot large changes.**

Coding with Warp is more effective if you:
- Ask the agent to make small, self-contained changes and commits
- Verify it is on the right track frequently
- Avoid the anti-pattern of trying to one-shot complex changes that require the agent to produce a lot of code
- Have the agent frequently write tests as it goes to force it to work incrementally
- Frequently look at the change in progress to make sure it's proceeding as it would if you were to write it by hand

### Code Review Process

- For now, you can use the Github UI or another PR review tool
- Eventually Warp will support this review flow more natively
- Have Warp make commits with good messages as it goes so you can revert if needed

### Understanding the Work

- Make sure you comprehend and understand what Warp is doing
- Ask for an explanation or plan before Warp starts making changes
- Iterate on the plan and ask for options and different approaches
- Make sure you get the big picture before it starts making diffs
- You can have Warp do a big change end-to-end to see what's possible, but expect to have to go back and break that up into comprehensible pieces post-hoc to meet quality standards

## Building Expertise

### Learning and Sharing Knowledge

- You will build instincts around what context to include and what types of prompting work best as you use Warp more to code
- Please share this knowledge with your team – coding by prompt is a totally new discipline and everyone has different instincts on how to do it well
- The team will learn faster together

### Advanced Features

Use the advanced features of Warp to improve your workflows:
- Use rules, prompts, and WD objects as persistent context
- Use "@" for tagging files
- Attach images for UI changes and debugging
- Experiment with different models
- Set up multiple copies of warp-internal and warp-server locally for effective multi-threading

## Feedback and Issue Reporting

**Report issues** – this is emphasized throughout:

If something isn't working well, report it to the appropriate #feedback- channel. This could include:
- Server errors or client bugs
- Agent going in circles, burning requests and wasting time
- Agent forgetting context
- Missing functionality that would help achieve the task
- High latency, AI acting dumb, mis-applied diffs, etc.

The first instinct on the feedback handling side should be to make evals for these issues.

## Advanced Setup

### MCP Servers for Context Retrieval

Set up MCP servers for easily retrieving context. Servers the team should all have:
- Sentry
- Linear
- Notion
- Slack
- (other context sources)

Warp is surprisingly capable at fixing panics and getting up to speed from Linear.

### Rules for Repeated Conventions

Take the time to set up Warp Drive rules for things you find yourself repeatedly instructing the agent to do:
- What linters to run
- Code conventions on unwrap/expect
- Import conventions
- Test location, etc.

### Prompts for Repeated Workflows

If you find yourself repeating a task, it's worth setting up a Warp Drive prompt for it:
- Generating commits
- Opening and editing PR descriptions
- Resolving merge conflicts
- And more
